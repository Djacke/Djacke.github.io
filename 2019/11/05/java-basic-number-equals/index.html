java基础-数值类型比较

2019-11-05
java
本文被浏览 次

如果第一眼就能想到答案，恭喜你，可以不用往下看了😂

Objects.equals(Long.valueOf(1), 1) = ？
Objects.equals(Long.valueOf(1), 1L) = ？
Objects.equals(Integer.valueOf(1), 1) = ？
Objects.equals(Integer.valueOf(1), Long.valueOf(1)) = ？

首先看看Objects.equals 的源码

再看看Long.valueOf 的源码,加载时初始化了cache数组-128 ～ 127，Integer类同理

  • LongCache.cache的初始化

接下来看看Long和Integer的equals方法

  • Long

  • Integer

两个类的equals方法几乎一样，但都调用了.xxxValue() 方法，此方法返回对象的非包装类型的常量值。其实我有点疑惑equals方法的入参的类型为什么不是当前类型，因为obj instanceof Integer这个判断就已经过滤了其他类型，所以感觉没必要入参类型为Object的

综上所述

Objects.equals(Integer.valueOf(1), Long.valueOf(1)) = false 无疑

我们再来看看java的自动装箱，1 作为实参传递给形如xxxMethod(Object obj)的方法时就会经过自动装箱，将基本类型int转换为Integer,1L 就会转为Long。
再回到文章开头的表达式，不难得出结果

false
true
true
false

我们不妨再看个例子

Long a = 10L;
Long b = 10L;
System.out.println(a == b);
Long c = 129L;
Long d = 129L;
System.out.println(c == d);

一定有人会产生困惑，解决办法只有一个，首先编译你的.java文件，然后javap -verbose ${youClassName},查看字节码，会发现如下的字节码片段

invokestatic #14; //Method java/lang/Long.valueOf:(J)Ljava/lang/Long;

所以10L 或者 129L 都会经过Long.valueOf创建Long的实例，问题就会又回到第一个例子的问题了，很容易得到如下结果

true //因为10L处于缓存区间之内，比较的是同一个对象的地址
false //129L超出缓存区间，初始化会创建连个对象，地址相异

纯属个人整理的点滴知识，欢迎批评指正。
